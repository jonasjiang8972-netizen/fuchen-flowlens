## 二、Phase 1 详细排期（0-3个月 · 补短板）

### 2.1 任务分解

| 序号 | 任务 | 版本 | 工时 | 依赖 | 交付标准 |
|------|------|------|------|------|----------|
| P1-1 | **免认证演示模式** | v0.5.0 | 0.5天 | 无 | 启动 `--demo` 即可展示全量 seed 数据，无需登录 |
| P1-2 | **BOLA 检测引擎上线** | v0.5.0 | 2天 | 已有引擎框架 | 模拟遍历 1000 个 ID 检出率 ≥85% |
| P1-3 | **认证失效/撞库引擎上线** | v0.5.0 | 2天 | 已有引擎框架 | 单 IP 5 分钟 50 账号检出 ≥90% |
| P1-4 | **BFLA 检测引擎上线** | v0.5.0 | 1天 | 已有引擎框架 | 管理端点越权检出 ≥90% |
| P1-5 | **敏感数据正则引擎** | v0.6.0 | 3天 | 无 | 身份证/手机号/银行卡/邮箱 4 类正则+校验位 |
| P1-6 | **脱敏核验引擎上线** | v0.6.0 | 2天 | P1-5 | 响应体脱敏缺陷自动标记 |
| P1-7 | **Bot 基础检测** | v0.6.0 | 3天 | 无 | JA3 指纹 + 请求间隔方差 + 爬虫识别 ≥70% |
| P1-8 | **Flink 流处理集成** | v0.7.0 | 5天 | 无 | Agent → Kafka → Flink → ClickHouse 链路打通 |

### 2.2 v0.5.0 具体工作包

**目标**：补齐 MVP 检测能力，降低演示门槛

```
v0.5.0 ─── 1 周（5人天）
├── 免认证演示模式（0.5天）
│   ├── platform: --demo 参数 → 跳过 JWT 中间件
│   ├── frontend: localStorage 无 token 时跳过 login
│   └── frontend: 全部使用 mock 数据回退
│
├── BOLA 检测上线（2天）
│   ├── engine/bola.go: 完善已建 Baseline 模型
│   ├── engine/bola.go: 添加 Isolation Forest 异常检测
│   ├── server: BOLA 告警自动写入 AlertService
│   └── frontend: 告警列表展示 BOLA 引擎产出
│
├── 认证失效引擎上线（2天）
│   ├── engine/auth_failure.go: 完善撞库特征模型
│   ├── engine/auth_failure.go: 添加 Token 熵值分析
│   ├── server: 认证告警自动写入 AlertService
│   └── frontend: 展示撞库/认证告警
│
└── BFLA 引擎上线（1天）
    ├── engine/bfla.go: 完善角色-接口矩阵
    ├── server: BFLA 告警自动写入
    └── frontend: 展示越权告警
```

### 2.3 v0.6.0 具体工作包

**目标**：补齐敏感数据 + Bot 检测

```
v0.6.0 ─── 1.5 周（8人天）
├── 敏感数据正则引擎（3天）
│   ├── engine/sensitive.go: 身份证正则+校验位
│   ├── engine/sensitive.go: 手机号正则
│   ├── engine/sensitive.go: 银行卡 Luhn 校验
│   ├── engine/sensitive.go: 自定义规则扩展
│   └── storage: 敏感字段持久化
│
├── 脱敏核验（2天）
│   ├── engine/masking.go: 脱敏模板匹配
│   ├── engine/masking.go: 实际返回 vs 预期脱敏
│   └── frontend: 脱敏缺陷看板增强
│
├── Bot 基础检测（3天）
│   ├── engine/bot.go: JA3/JA4 指纹采集
│   ├── engine/bot.go: 请求间隔方差分析
│   ├── engine/bot.go: 资源加载特征检测（无CSS/JS）
│   └── frontend: Bot 告警展示
│
└── Agent: 采集层 Bot 指纹上报（与 engine 并行）
    ├── agent: eBPF 采集 TLS 握手指纹
    └── agent: HTTP 请求头指纹
```

### 2.4 v0.7.0 具体工作包

**目标**：打通全链路数据管道

```
v0.7.0 ─── 1 周（5人天）
├── Flink 流处理作业（3天）
│   ├── Flink Job: Kafka → Normalizer → ClickHouse
│   ├── Flink Job: 窗口聚合（5min/1h/24h）
│   └── Flink Job: 基线统计写入
│
├── ClickHouse 集成（1天）
│   ├── storage/clickhouse.go: 批量写入
│   ├── storage/clickhouse.go: 时序查询
│   └── deploy: ClickHouse 表结构
│
└── 端到端验证（1天）
    ├── Agent → Kafka → Flink → CH → Platform
    └── 10 万 QPS 压测验证
```

## 六、风险与应对

| 风险 | 等级 | 概率 | 应对方案 |
|------|------|------|----------|
| Flink 集成复杂度超预期 | 🔴 高 | 40% | MVP 先用 Kafka + 自研聚合器替代，V1.0 再迁移 Flink |
| Bot 检测研发周期过长 | 🟡 中 | 30% | 先上 JA3 指纹+简单统计，ML 模型延后 |
| 信创适配成本超出预算 | 🟡 中 | 30% | 先用容器化规避操作系统差异，数据库用 PG 兼容模式 |
| OWASP 检测线下指标达标但线上误报高 | 🟡 中 | 50% | 预留 2 周调优窗口 + 误报反馈闭环机制 |
