## 一、项目定位与命名哲学

### 拂尘（The Whisk）

纯旁路（Bypass）部署，如清风拂面。在**零业务侵入、零性能损耗、零改造风险**（三零原则）下，悄悄拂去 API 中的恶意威胁与敏感数据泄露隐患，保障生产环境纯净。

### FlowLens（流量棱镜）

Flow 代表流不息的 API 流量，Lens 代表高精度透视。像多维棱镜一样，将混杂、加密的数字化流量深层拆解，隐蔽威胁无所遁形。

### 内部代号

API Sentinel

### 核心定位

以旁路流量镜像与云原生日志分析为核心手段，构建"持续可见、深度检测、开放联动"的 API 资产与风险治理平台——重点解决"看不见、看不全、看不深"的问题，检测到风险后通过标准接口与网关/WAF/SOAR联动完成处置闭环。

## 二、行业背景与核心痛点

### 三大瓶颈

| 痛点 | 描述 |
|------|------|
| **投鼠忌器** | 串联部署存在单点故障风险，高并发下深层解析带来毫秒级延迟，业务无法接受 |
| **资产盲区** | 研发迭代快，"影子 API"、"僵尸接口"泛滥，安全团队对家底模糊不清 |
| **逻辑漏洞** | 越权访问（BOLA/BFLA）、慢速拖库等业务逻辑攻击没有典型特征码，传统 WAF 无法拦截 |

### 合规驱动

《数据安全法》《个人信息保护法》、JR/T 0185-2020《商业银行应用程序接口安全管理规范》、GB/T 39276-2020 等法规标准对 API 数据流转提出明确合规要求，推动 API 安全从"可选项"变为"必选项"。

### 目标客户

优先级从高到低：金融（银行/保险/证券）→ 政务与关基 → 医疗 → 能源/运营商 → 中大型互联网

## 三、设计原则与架构总览

### 设计原则

1. **旁路优先**：部署不改变现网请求路径
2. **检测与处置解耦**：专注深度检测与风险研判，处置联动网关/WAF/SOAR 执行
3. **开放集成**：所有检测结果、资产数据通过标准 API/Webhook 开放
4. **统一采集**：单一采集工具适配所有环境（eBPF/SPAN/Gateway Log/VPC Flow）
5. **状态可查**：采集器与平台双向心跳，断联即告警
6. **适配优先**：协议、网关、云平台、部署形态广泛兼容
7. **安全自证**：平台自身不低于所保护对象的安全等级

## 四、统一采集层（FlowLens Agent）

### 4.1 设计目标

- **一个二进制**：自动探测环境，切换采集模式
- **全场景覆盖**：K8s/物理机/VM/公有云/边缘
- **平台可观测**：双向心跳、实时指标、远程管理
- **断联可感知**：60s 无心跳触发告警

### 4.2 自动探测逻辑

```
启动
  │
  ├─ 是 K8s？
  │   ├─ 有 Cilium？ → Mode: eBPF (Cilium hook)
  │   └─ 无 Cilium？ → Mode: eBPF (通用 CO-RE)
  │
  ├─ 有 DPDK + SPAN？ → Mode: DPDK_SPAN
  │
  ├─ 有网关日志目录？ → Mode: Gateway_Log
  │
  ├─ 有云 API 凭证？ → Mode: VPC_Flow_Pull
  │
  └─ 兜底 → Mode: pcap (libpcap)
```

### 4.3 采集模式详解

#### eBPF 模式（云原生/推荐）

- 基于 CO-RE eBPF 程序
- 挂载 `tcp_sendmsg` / `tcp_recvmsg`
- 在 Service Mesh sidecar mTLS 解密点采集 → 天然明文
- 支持过滤端口：80, 443, 8080, 8443
- 可选 uprobe SSL_read/SSL_write（tls 解不开时）

#### SPAN/DPDK 模式（传统环境）

- DPDK 内核旁路，零拷贝
- 物理 TAP 或交换机 SPAN 端口
- 适用于：TLS 已卸载后的内网流量

#### 网关日志模式（快速接入）

- File tail 方式读取 Kong/APISIX/Nginx/Envoy 日志
- 支持 JSON / Nginx / Envoy 格式
- 零侵入，最快上线

#### VPC Flow 模式（SSRF 增强）

- 定时拉取云厂商 VPC Flow Logs
- 仅用于 FR-DET-007 出向连接检测
- 配合 eBPF 做增强判定

#### pcap 兜底模式

- 无特权、无 DPDK、无 eBPF 时的最后手段
- libpcap 抓包，吞吐较低

### 4.5 心跳与管理通道

```
Agent ───────────────────────────────── 平台管控面
  │                                        │
  │  WebSocket/mTLS 长连接                   │
  │                                        │
  │── Heartbeat(10s) ──────────────────►  │
  │    { agent_id, status, metrics }       │
  │                                        │
  │◄── ConfigPush (热更新) ──────────────  │
  │    { kafka_brokers, filter_rules }     │
  │                                        │
  │◄── Command ─────────────────────────  │
  │    { restart, stop, upgrade }          │
  │                                        │
  │── LogUpload ───────────────────────►  │
  │    { level: error, msg: "..." }       │
```

**超时判定**：

| 时间 | 动作 |
|------|------|
| 心跳延迟 > 30s | UI 黄色警告 |
| 心跳丢失 > 60s | 触发告警 |
| 心跳丢失 > 5min | 升级告警至安全负责人 |

## 五、数据管道层（The Conduit）

### 5.1 组件

| 组件 | 选型 | 用途 |
|------|------|------|
| 消息队列 | Kafka | 流量削峰、持久化回溯 |
| 流处理 | Apache Flink | 流式重组、聚合、基线计算 |
| 存储 | ClickHouse | 流量日志、查询分析 |
| 搜索 | Elasticsearch | 威胁检索、全文搜索 |
| 元数据 | PostgreSQL | 资产、配置、工单 |
| 缓存 | Redis | 会话、热数据、分布式锁 |

### 5.2 数据流

```
Agent → Kafka (raw-api-events topic)
          │
          ├── Flink Job 1: 实时归一化 → Kafka (normalized-events)
          │
          ├── Flink Job 2: 资产发现写入 → PostgreSQL + ClickHouse
          │
          ├── Flink Job 3: 基线统计写入 → Redis + ClickHouse
          │
          └── Flink Job 4: 原始数据采样 → ClickHouse (TTL 90天)
```

## 六、分析引擎层（The Lens Engine）

### 6.1 引擎组成

- **规则引擎**：基于 YARA/Suricata 风格模式匹配
- **UEBA 引擎**：无监督学习构建行为基线
- **AI/ML 引擎**：异常检测、聚类、NLP 敏感信息识别
- **关联引擎**：跨请求上下文关联、攻击链还原

### 6.2 检测流水线

```
normalized-event
      │
      ▼
┌─────────────┐    命中规则 → 直接输出告警
│  规则引擎    │
└──────┬──────┘
       │ 未命中
       ▼
┌─────────────┐    基线偏离 → 输出风险信号
│  UEBA 引擎   │
└──────┬──────┘
       │ 未偏离
       ▼
┌─────────────┐    AI 异常 → 输出疑似告警
│  AI/ML 引擎  │
└──────┬──────┘
       │ 正常
       ▼
   更新基线库，放行
```

## 八、平台管控面（含采集器管理）

### 8.1 采集器注册流程

```
Agent 启动 → mTLS 握手 → 注册请求 (环境/模式/容量)
    → 平台签发证书 → 建立 gRPC 长连接
    → 下发初始配置 → Agent 开始采集
```

### 8.2 采集器状态模型

```
注册中 → 在线 → 采集运行中
                   ├─ 心跳延迟 → 异常告警
                   ├─ 心跳丢失 → 离线告警
                   └─ 配置更新 → 热加载
```

### 8.3 远程管理能力

| 能力 | 说明 |
|------|------|
| 配置热更新 | Kafka地址/过滤规则/采样率 实时生效 |
| 远程启停 | 平台下发启停指令 |
| 日志级别动态调整 | 运行时切换 debug/info/warn |

## 十四、风险与应对

| 风险项 | 等级 | 应对措施 |
|--------|------|----------|
| TLS/QUIC 加密导致不可见 | 高 | 优先 TLS 卸载点部署 + eBPF uprobe 兜底 |
| 云厂商原生集成挤压 | 高 | 聚焦行业化合规 + 深度检测差异化 |
| 误报率过高致弃用 | 中 | 建立误报反馈闭环 + 持续优化基线 |
| 全量流量存储成本高 | 中 | 元数据+采样存储策略 + 敏感字段单独加密 |
| 新型 API 形态（MCP/Agent） | 中 | 建立威胁情报跟踪机制 |
| 信创环境兼容性 | 中 | 提前建立信创实验室 |
| 采集器断联未被感知 | **已解决** | 双向心跳 + 60s 超时告警 + 自动升级 |
| 多工具适配复杂度高 | **已解决** | 统一 FlowLens Agent 自动探测模式 |

| 术语 | 释义 |
|------|------|
| 旁路 (Bypass) | 通过流量镜像/TAP/SPAN/Sidecar/eBPF 采集副本，不改变原始请求路径 |

| UEBA | 用户与实体行为分析 |
| SOAR | 安全编排、自动化与响应 |
| 影子 API | 未纳入统一管理的在用 API |
| 僵尸 API | 已下线但仍可访问的 API |
| CO-RE | Compile Once, Run Everywhere（eBPF 跨内核兼容方案） |
| mTLS | 双向 TLS 认证 |
| uprobe | 用户态动态追踪探针（eBPF 子功能） |
| JA3/JA4 | TLS 握手指纹算法 |
